# Context Notes — ShotStage 고도화 버전

이 문서는 작업 중 내려진 결정과 그 이유를 기록한다. 다음 세션(사람이든 에이전트든)이 왜 이렇게 설계했는지 다시 유추하지 않아도 되도록 계속 이어서 기록한다.

## 원본 사이트 분석 관련
- 원본(https://shotstage.vercel.app/)은 클라이언트 사이드 렌더링 SPA라 web_fetch로는 페이지 타이틀 외 콘텐츠를 가져올 수 없었다. 사용자가 제공한 스크린샷 3장(메인 화면, 프롬프트 패널, 결과 모달)을 근거로 기능을 역추적했다.
- 원본 소스코드는 확보하지 못했으므로, 기존 기능도 처음부터 재구현해야 한다 (checklist.md Phase 1~3).
- 결과 모달의 프롬프트 문구에 "Use the attached blocking frame only as the camera reference... Ignore its plain placeholder surfaces and colours"라는 문장이 있다. 즉 3D 뷰포트는 의도적으로 무채색 placeholder로 남겨두고, 실제 디테일은 AI 생성 단계에서 채우는 설계다. 이 설계 철학이 이후 DOF 기능을 제외하는 근거가 되었다.

## 범위 결정
- 타겟 사용자: 개인 크리에이터(1인 작업). 팀 협업/계정/서버 저장은 이번 범위에서 제외.
- 우선순위로 선택된 두 영역: (1) 카메라 고급 기능, (2) 캐릭터·환경 에셋 라이브러리.
- 제외된 영역(향후 별도 PRD 대상): 멀티 샷/스토리보드 관리, AI 모델·생성 연동 확장(여러 모델·프롬프트 히스토리·직접 생성), 팀 협업.

## 자체 점검을 통해 축소한 항목 (중요 — 재검토 시 참고)
최초 초안에는 아래 세 가지가 더 있었으나, "원본은 단일 화면·한 번에 하나 확정하는 흐름"이라는 판단 기준으로 자체 점검 후 제외했다.

1. **심도(DOF) 실시간 블러 렌더링** — 제외 이유: 3D 뷰포트가 애초에 "무시하라"고 명시하는 placeholder라, 렌더링 비용 대비 실익이 없음. React Three Fiber post-processing 파이프라인 추가 비용도 상당함.
2. **무빙샷 시작/끝 2단계 프레임 캡처** — 제외 이유: 프레임을 2장 관리하는 완전히 새로운 워크플로우라 원본의 "한 번에 하나 확정" 철학과 충돌. 대신 프레임 1장 + 무브 프리셋(방향/강도/지속시간)만 추가하는 방식으로 축소.
3. **에셋 라이브러리 별도 관리 화면** — 제외 이유: 원본에 없던 "페이지" 개념을 추가하는 것이라 과함. 대신 입력창 옆 드롭다운 안에서 inline 삭제(hover 시 × 아이콘)와 덮어쓰기 저장으로 대체.

**주의**: 이 세 가지를 나중에 다시 추가하고 싶다는 요청이 있으면, 이 판단 기준(원본의 단일 화면·즉시 확정 흐름 유지)에 부합하는 대안적 구현 방식을 먼저 고민한 뒤 추가할 것. 무조건 원안(2단계 캡처 등)으로 되돌리지 말 것.

## 사용성 재검토 후 보완한 항목
- 무빙샷 강도(약함/보통/강함)를 미리보기 없이 감으로 선택해야 한다는 문제가 사용 시나리오 점검 중 발견됨.
- 해결책으로 "미리보기 프레임 재도입"(2단계 캡처 부활)과 "설명 툴팁 추가" 두 가지를 검토했고, 사용자가 가벼운 쪽(툴팁)을 선택함.
- 각 강도 버튼에 짧은 설명 문구를 툴팁으로 추가하기로 결정 (PRD 5.3절, checklist Phase 5).

## 기술 스택 결정 및 근거
- 저장소: IndexedDB(브라우저 로컬 저장)만 사용. 서버/DB/인증 없음. 근거: 개인 크리에이터 타겟 + MVP 범위 + 협업 비목표.
  - 주의: 추후 여러 기기 동기화나 팀 협업이 필요해지면, 이 저장 방식부터 재설계가 필요함.
- 프레임워크: Next.js(App Router) + React Three Fiber/drei + Zustand + Tailwind CSS. 원본도 유사한 스택으로 추정되나 확인된 사실은 아님(가정).
- 배포: GitHub 저장소 → Vercel (사용자가 처음부터 명시한 요구사항).

## 구현(Phase 0~6) 진행 중 내려진 세부 결정
- **개발 환경 제약**: 이 작업 환경에는 헤드리스 브라우저가 없어, 실제 렌더링 화면을 스크린샷으로 찍어 원본과 픽셀 단위로 비교하는 시각적 검증을 하지 못했다. `npm run build`와 `npm run lint` 통과만 확인한 상태. 사용자가 로컬에서 `npm run dev`로 실행해 육안 확인을 하는 과정이 필요하다.
- **카메라 컨트롤 구조**: 3D 카메라의 실제 위치/각도(yaw, pitch, position)는 React 상태가 아니라 `CameraRig` 내부의 ref로 관리하고, three.js 카메라 객체를 `useFrame` 안에서 직접 mutate하는 R3F의 표준 패턴을 따랐다. HUD에 표시할 거리/뷰 방향 정보만 10프레임에 한 번씩 zustand 스토어로 던진다 (매 프레임 리렌더 방지).
  - 이로 인해 Next.js 16의 React Compiler 계열 ESLint 규칙(`react-hooks/immutability`)이 오탐(false positive)을 일으켜서, `src/components/viewport/**` 경로에 한해 해당 규칙을 껐다 (`eslint.config.mjs` 참고). R3F를 쓰는 이상 불가피한 예외이며, 다른 경로에는 규칙이 그대로 적용된다.
- **마네킹 정면 방향 규약**: 마네킹의 눈(얼굴)은 로컬 좌표계 기준 +Z 방향을 향하도록 배치했다. 샷 타입 프리셋의 카메라는 피사체의 +Z 쪽(정면)에서 바라보도록 스냅한다. Front/Back/Side 판정은 "피사체→카메라 벡터"와 "피사체 정면 벡터"의 내적으로 계산한다 (0.5 이상이면 front, -0.5 이하면 back, 그 사이는 side).
- **Bird's-eye 뷰 구현 방식**: 별도의 카메라 상태를 두지 않고, 기존 fly 카메라의 위치/시점(ref)은 그대로 둔 채 렌더링되는 프레임에서만 상공 시점으로 오버라이드한다. 그래서 Shot view로 되돌아가면 이전 시점이 그대로 복원된다.
- **프레임 캡처 방식**: `Canvas`의 `gl` 옵션에 `preserveDrawingBuffer: true`를 켜고, Generate 클릭 시 store의 `frameCaptureRequestId`를 증가시켜 Canvas 내부의 `FrameCapture` 컴포넌트가 `gl.domElement.toDataURL()`로 캡처하도록 구현했다.
- **AI 모델 드롭다운**: PRD에서 "AI 모델 연동 확장"은 명시적으로 이번 범위 밖이라고 했지만, 원본 스크린샷에도 이미 모델 선택 드롭다운(`GPT Image 2`)이 있었으므로 이건 신규 기능이 아니라 기존 UI 재현으로 판단해 그대로 넣었다. Image 모드에는 `GPT Image 2 / Nano Banana Pro / Flux`, Video 모드에는 `Higgsfield / Runway / Kling`을 정적 목록으로 넣었을 뿐, 실제 어떤 모델과도 API 연동은 하지 않는다 (프롬프트 텍스트만 생성).
- **IndexedDB 스키마**: PRD 6.4절 설계 그대로 `src/lib/db.ts`에 구현. 이름이 같은 프리셋을 저장하면 기존 id를 재사용해 덮어쓰는 방식으로 "이름 변경 없이 덮어쓰기 저장"을 구현했다 (PromptPanel의 `handleSaveCharacter` 등에서 `items.find(p => p.name === name)`으로 기존 항목을 찾음).
- **참조 이미지 용량**: 업로드된 이미지를 리사이즈/압축하지 않고 원본 그대로 data URL로 저장하고 있다. PRD 9절에 남겨둔 오픈 이슈 그대로 미해결 상태이며, 실사용 중 용량 문제가 확인되면 리사이즈 로직을 추가해야 한다.

## 확인되지 않은 가정 (오픈 이슈, PRD 9절과 동일)
- 원본이 mm ↔ FOV(도) 환산 로직을 이미 갖고 있는지 미확인. UI에 두 단위가 혼용되어 있어(FOV 슬라이더가 도 단위, 라벨은 mm 단위) 있다고 가정하고 렌즈 프리셋을 설계함.
- 캐릭터/환경 참조 이미지를 IndexedDB에 저장할 때 브라우저별 용량 제한을 고려해 리사이즈/압축이 필요할 수 있음. 아직 구체적인 리사이즈 기준(최대 해상도 등)은 정하지 않음 — 구현 시점에 결정 필요.
